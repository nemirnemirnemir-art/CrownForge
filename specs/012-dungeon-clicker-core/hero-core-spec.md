## Цель

Вынести героев в отдельный модуль "ядро героев".

Сейчас в игре есть герои, которых мы "нанимаем" (они дают пассивный урон/клики).

Нужно:
- перестроить это так, чтобы герои были объектами с параметрами (hp, damage, level, name, icon),
- больше не давали пассивный урон напрямую, а складывались в список героев,
- этот список дальше будет использоваться UI-панелью и боевой системой (в других спеках).

В этом спеκе НЕ делаем UI и бой, только данные и функции.

Используй тот же язык/стек, что уже используется в проекте для игровой логики (никаких новых фреймворков).

## Модель героя

Нужен отдельный модуль heroCore (название можешь подобрать под стиль проекта), где будет описана структура героя и функции работы с героями.

**Архитектура**: Модуль `HeroCore` реализуется как автолоад (singleton), аналогично `GameManager`. Хранит список всех героев внутри и предоставляет публичный API для работы с ними.

### Структура hero

Hero представлен как Dictionary со следующими полями:

- `id: String` — уникальный идентификатор героя (строка).
- `name: String` — имя героя (берётся случайно из списка имён ниже).
- `level: int` — текущий уровень, минимум 1.
- `baseHp: float` — базовое HP на 1 уровне (по умолчанию 10.0).
- `baseDamage: float` — базовый урон на 1 уровне (по умолчанию 1.0).
- `currentHp: float` — HP на текущем уровне (рассчитанное значение).
- `currentDamage: float` — урон на текущем уровне (рассчитанное значение).
- `iconId: String` — идентификатор иконки/спрайта для этого героя (например, "swordsman", "archer", "warrior_woman").
- `isUnlocked: bool` — флаг, что герой получен игроком и должен отображаться в списке.

Список всех героев хранится в `HeroCore` как `Array[Dictionary]` или `Dictionary` с ключами-ID для быстрого доступа.

## Формулы статов / левелап

### Базовые значения

При создании героя:
- `level = 1`
- `baseHp = 10`
- `baseDamage = 1`

### Формулы

```
currentHp(level)     = baseHp     + (level - 1) * 1
currentDamage(level) = baseDamage + (level - 1) * 1
```

То есть каждый новый уровень даёт +1 HP и +1 урона.

### Функции

Нужна утилита (функция) для пересчёта:
- `recalcStats(hero)` — обновляет currentHp и currentDamage на основе baseHp, baseDamage и level.

Функция левелапа:
- `levelUpHero(heroId, levels = 1)`:
  - увеличивает level на levels (по умолчанию 1),
  - вызывает recalcStats(hero).

Источник опыта/валюты для повышения уровня здесь НЕ реализуем, только сам переход уровня и пересчёт статов.

## Функции heroCore

Сделать модуль, который инкапсулирует работу с героями. Примерный интерфейс (подстрой под свой язык/архитектуру):

- `createHero(iconId: String) -> Dictionary` — создаёт нового героя:
  - генерирует уникальный `id` в формате "hero_N", где N — инкрементальный счетчик (начинается с 1),
  - выбирает случайное `name` из HERO_NAMES (по возможности без повторов),
  - ставит `level = 1`,
  - `baseHp = 10.0`, `baseDamage = 1.0`,
  - `isUnlocked = true`,
  - `iconId = переданный параметр`,
  - вызывает `recalcStats()` для расчета `currentHp` и `currentDamage`,
  - добавляет героя во внутренний список,
  - возвращает Dictionary с данными героя.

- `getAllHeroes() -> Hero[]`
- `getUnlockedHeroes() -> Hero[]`
- `getHeroById(id) -> Hero | null`
- `levelUpHero(heroId, levels = 1)`
- `addHero(hero: Hero)` / внутренний список, если нужно.

## Интеграция с сохранениями

Добавить сохранение/загрузку списка героев в существующую систему сейвов.

### Формат сохранения

Если структура сейва уже есть:
- добавить новый блок `"heroes": [...]` в JSON, не ломая старые поля.
- Каждый герой сохраняется как Dictionary с полями: `id`, `name`, `level`, `baseHp`, `baseDamage`, `currentHp`, `currentDamage`, `iconId`, `isUnlocked`.

### Загрузка и миграция

При загрузке:
1. **Если в сохранении есть блок `"heroes"`**:
   - Загрузить всех героев из массива.
   - Восстановить все поля: `level`, `baseHp`, `baseDamage`, `currentHp`, `currentDamage`, `name`, `iconId`, `isUnlocked`.
   - Прогнать `recalcStats()` для каждого героя, чтобы значения были консистентны.

2. **Если в сохранении есть старый блок `"passive_units"` (миграция)**:
   - Для каждого пассивного юнита (swordsman, archer, warrior_woman):
     - Если `level > 0`, создать соответствующее количество Hero-объектов через `createHero(iconId)`.
     - Для каждого созданного героя установить `level` равным уровню пассивного юнита.
     - Вызвать `recalcStats()` для каждого героя.
   - После миграции удалить блок `"passive_units"` из сохранения (не сохранять его в будущем).

3. **Если нет ни `"heroes"`, ни `"passive_units"`**:
   - Начать с пустого списка героев.

## Интеграция с текущей логикой (важно)

Важно: сейчас часть героев может давать пассивный урон/клики. В рамках ЭТОГО спека:

**Стратегия замены**: Полностью заменить существующую систему `passive_units` на новую систему Hero.

В местах, где сейчас создаются/нанимаются герои (например, "герои слева, дающие пассивный урон"):
- Удалить `passive_units` из `GameManager`.
- При покупке героя (кнопки "LVL UP" для swordsman, archer, warrior_woman):
  - вместо увеличения уровня в `passive_units` и расчета пассивного урона,
  - создать нового героя через `heroCore.createHero(iconId)` и сохранить его в список героев.
  - `iconId` должен соответствовать типу героя: "swordsman", "archer", "warrior_woman".
- Пассивный урон от этих героев временно отключить (удалить вызовы `get_total_passive_cps()` и связанную логику).

Никакой UI, панелей, кнопки "в бой" в этом спеκе не делаем.

Никакой боевой логики для героев здесь НЕ нужно — только данные и функции.

## Нефункциональные требования

- Не ломать основной клик-луп и существующую экономику, кроме отключения пассивного урона от героев (если он был).
- Все изменения по героям должны быть локализованы в новом модуле heroCore + в местах, где они создаются/нанимаются.
- Код должен быть написан в стиле и архитектуре проекта (те же паттерны, та же структура файлов).
